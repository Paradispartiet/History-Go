# Batch 27 — wrong-prefix decision audit

**Dato:** 2026-05-31

## Formål og avgrensning

Dette er en read-only beslutningsrapport etter Batch 26. Målet er å klassifisere alle 304 wrong-prefix `emne_id`-forekomster slik at senere Batch 28+ kan avgjøre hva som skal beholdes som tverrfaglighet, byttes, fjernes, løftes til place-category-vurdering eller behandles som prefixpolicy/legacy-konvensjon.

**Ingen datafikser er gjort.** Rapporten endrer ikke `data/places/**`, `data/fag/**`, `data/places/places_index.json`, manifest, UI, scripts, bilder eller emne-id-er.

## Kommandoer kjørt

- `npm run places:emner:check`
- `npm run places:index:check`
- `npm run health:places`
- `python3 /tmp/make_batch27_report.py` — read-only parsing av aktive place-filer, canonical emne-filer, domain matrix/pensum-filer og health-prefix-reglene for å lage denne rapporten.
- `git status --short`

NPM skrev `npm warn Unknown env config "http-proxy"...` før scriptkjøringene. Varslet påvirket ikke exit code.

**Anbefaling for fremtidig tverrfaglighet:** innfør en redaksjonell markering/allowlist per place-emne-kobling eller per prefix-par før health-scriptet eventuelt demper warnings. Markeringen bør angi om koblingen er primær, sekundær/tverrfaglig eller legacy-policy.

## Mulige faktiske feil og senere småbatcher

- Places med mange wrong-prefix-koblinger bør prioriteres først, fordi de kan skjule category-drift eller overmapping.
- Prefix-par der actual fagfamilie virker langt fra place-kategorien bør vurderes manuelt før endring.
- Denne batchen fant ingen trygg automatisk sletting: alle ID-er er canonical, og de største gruppene har plausible tverrfaglige eller policybaserte forklaringer.

### Kandidater til senere småbatcher

| Batch-kandidat | Innhold | Hvorfor |

| Batch 28A policy | em_naer_* legacy-prefix | Avklar om health-scriptet skal tillate em_naer_* for naeringsliv eller om data skal migreres etter kontrollert mapping. |
| Batch 28B tverrfaglighet | by→populaerkultur, by→film_tv, by→kunst | Marker/allowlist bevisste tverrfaglige koblinger, særlig i Oslo by-filene. |
| Batch 28C urbane naturkoblinger | natur→by | Vurder om urban natur/elve- og parkruter skal få eksplisitt tverrfaglig byromstillatelse. |
| Batch 28D litteratur i byrom | litteratur→by | Vurder statuer, minnesteder, bibliotek og bokhandler som litteratur + byrom uten å endre category blindt. |
| Batch 28E manuell overmapping | places med flest wrong-prefix | Gå gjennom topp place-id-er én og én for å fjerne svake koblinger eller bytte til category-matchende canonical emner. |

## Hva er trygt å fikse automatisk?

- **Ingen emne-koblinger bør endres automatisk i denne fasen.**
- Det tryggeste automatiske tiltaket senere er eventuell health-policy/allowlist for dokumentert legacy-prefix, men også det bør være egen policy-PR.
- Blind prefix-rewrite er ikke trygt: `em_by_*`, `em_pop_*`, `em_his_*` osv. er selvstendige canonical emner, ikke aliaser til category-matchende ID-er.

## Hva krever redaksjonell vurdering?

- Om tverrfaglige koblinger skal være synlige som sekundære koblinger eller fortsatt gi health-warning.
- Om `em_naer_*` og eventuelle psykologi-legacy-prefix skal tillates av policy.
- Om enkelte places har feil category, eller bare har sterke sekundærkoblinger.
- Om places med mange wrong-prefix har for bred emne-mapping.

## Anbefalt Batch 28

Anbefalt neste PR: **Batch 28A — policyavklaring for `em_naer_*` og wrong-prefix allowlist-modell**. Den bør fortsatt være liten og eksplisitt: enten dokumentere legacy-prefixpolicy i health-scriptet, eller lage en kontrollert migreringsplan med mapping fra `em_naer_*` til `em_naering_*` der det faktisk finnes én-til-én-ekvivalens.

Deretter bør Batch 28B ta de største tverrfaglige prefix-parene som redaksjonell allowlist/markering, ikke data-rewrite.
